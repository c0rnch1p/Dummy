	DESCRIPTION: Obsidian is a robust knowledge management tool enabling the
	 composition, structuring, and linking of notes effectively. Its used widely for
	 its support of biodirectional linking and development of atomic notes, which
	 are distinct concepts or units of information linkable through references. A
	 key feature of Obsidian is the graph view, also referred to as the knowledge
	 graph or second brain. The interface utilizes notes as nodes interconnected
	 visually, outlining a network of connections within the note database,
	 otherwise known as a vault. Additionally, Obsidians capabilities can be
	 extended with various community plugins and unofficial packages including the
	 Text Generator plugin, which integrates AI powered text generation directly
	 into Obsidian. This feature can assist in brainstorming, expanding on ideas and
	 generating content based on prompts within their notes environment.

	COMMON USAGE
	 DIRECTORY STRUCTURE
	   # Integrating Obsidian smoothly into a workflow can be overwhelming if the
	   # fundamental idea behind the usage of the program isnt fully understood.
	   # Files other than notes created by the user will inevitably end up inside
	   # the vault because of a setting for a plugin or some other thing. Below is a
	   # tried and tested obsidian vault structure that will leave no notes lying
	   # loose in the top level directory and a clean graph view, assuming that
	   # everything is linked correctly
	  01_CURRENT # Folder for top priority notes
	  02_DUMP # Loose notes and ideas
	  03_LINKS # Folder for storing loose link notes
	  04_TAGS # Folder for storing tags
	  05_TEMPLATES # Obsidian templates folder
	  06_LAYOUTS # Literal directory layouts
	  07_AUDIO # Folder for audio files
	  08_VISUAL # Folder for image and video files
	  09_SOURCE # Source content and external resources folder
	 YAML FRONT MATTER
	   # In Obsidian, YAML is natively supported for use within the properties
	   # section at the top of a note to add metadata that caters to various
	   # requirements such as searching, categorizing, and plugin functionality. The
	   # properties section is defined by a '---' at the start and end of a section
	  --- # YAML delineator
	  title: "Obsidian" # Create a title property
	  tags: ["layouts_development_dummyfiles"] # Create a tag property list
	  todo: ["Yaml usage", "Dataview specs"] # Create a todo list for the file
	  stat: ["incomplete"] # Give the file a status
	  aliases: ["ob"] # Create an alias for the file
	  --- # YAML delineator
	   # YAML properties should adhere to correct syntax with keys followed by a
	   # colon and then the value. Strings can be quoted, while numbers and booleans
	   # are typically not. Indentation is not required in this simple key-value
	   # pair structure, but double space indentations are also supported for longer
	   # lists and nested elements
